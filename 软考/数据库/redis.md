
## # redis怎么实现高可用？


#### 1.主从模式
1. 主从复制包括：全量复制和增量复制两种，一般第一次启动连接master，或者认为是第一次连接，就采用全量复制。全量同步完成后，master上的数据，如果再发生更新，就会触发增量复制。

2. 当master节点发生数据增减时，就会触发 replicationFeedSlaves() 函数，接下来在master节点上调用的每一个命令会使用 replicationFeedSlaves() 来同步到slave节点。执行此函数之前呢，master节点会判断用户执行的命令是否有数据更新，如果有数据更新的话，并且slave节点不为空，就会执行此函数。这个函数作用就是：把用户执行的命令发送到所有的slave节点，让slave节点执行。

3. redis 的 replicaof 命令用于设置一个redis服务器（从服务器）复制另一个redis服务器（主服务器）
4. SAVE：同步执行，直接在主进程调用 rdbsave函数，阻塞所有客户端请求直至RDB文件生成完成
5. BGSAVE：异步执行，通过fork子进程处理RDB文件生成，主进程继续响应客户端请求，仅fork阶段有短暂阻塞。
6. repl_backlog是Redis主从复制机制中用于记录命令日志和offset的关键组件，主要用于实现数据同步。核心功能 repl_backlog 通过环形数组记录主节点master执行的写命令及其offset，确保从节点在断线后能通过该日志恢复数据同步。当主节点写入新数据时，旧日志会被覆盖，但未同步的增量数据会保留在日志中，直到从节点完成同步。

7. 全量同步：首次连接时，主节点通过RDB文件和repl_backlog将全部数据发送给从节点。
8. 增量同步：后续通过repl_backlog 记录的写入命令更新从节点数据，实现增量复制。

#### 2.哨兵模式（sentinel）

1. 故障切换过程：假设主服务器宕机，哨兵1检测到这个结果，系统并不会马上进行 failover 过程，仅仅是哨兵1主观的认为主节点不可用，这个过程称为“主观下线”。当其它的哨兵也检测到主节点不可用，并且数量达到一定值时，那么哨兵之间会进行一次投票，投票结果由一个哨兵发起，进行failover操作。切换成功后，就会通过发布订阅模式，让各个哨兵把自己监控的从节点实现切换主机，这个过程称为客观下线。这对客户端来说一切都是透明的。

#### 3.集群模式（cluster）

1. Redis Cluster 集群通过Gossip协议进行通信，节点之间不断交换信息，交换的信息内容包括节点出现故障、新节点加入、主从节点变更信息、slot信息等等。常用的Gossip消息分为4种，分别是：ping、pong、meet、fail。
  - meet消息：通知新节点加入。消息发送者通知接收者加入到当前集群，meet消息通信正常完成后，接收节点会加入到集群中并进行周期性的ping、pong消息交换。
  - ping消息：集群内交换最频繁的消息，集群内每个节点每秒向多个其它节点发送ping消息，用于检测节点是否在线和交换彼此状态信息。
  - pong消息：当收到ping、meet消息时，作为响应消息回复给发送方确认消息正常通信。pong消息内部封装了自身状态数据。节点也可以向集群内广播自身的pong消息来通知整个集群对自身状态进行更新。
  - fail消息：当节点判定集群内另一个节点下线时，会向集群内广播一个fail消息，其它节点收到fail消息之后把对应节点更新为下线状态。

2. 特别的，每个节点是通过集群总线（cluster bus）与其它的节点进行通信的，通讯时，使用特殊的端口号，即对外服务端号加10000，例如如果某个node的端口号是6379，那么它与其它nedes通信的端口是16379。nodes 之间的通信采用特殊的二进制协议。

3. cluster 的分布式算法是：Hash Slot插槽算法。

4. Hash Slot插槽算法（软考大概率不会考具体算法内容）：把整个数据库分为16384个slot（槽），每个进入Redis的键值对，根据key进行散列，分配到这16384个插槽中的一个。使用的哈希映射也比较简单，用CRC16算法计算出一个16位的值，再对16384取模。数据库中的每个键都属于这16384个槽中的一个，集群中的每个节点都可以处理这16384个槽。集群中的每个节点负责一部分的hash槽，比如当前集群有A、B、C个节点，每个节点上的哈希槽数=16384/3。

5. 集群中，需要确保16384个槽对应的node都正常工作，如果某个node出现故障，它负责的slot也会失效，整个集群将不能工作。因此为了保证高可用，集群引入了主从复制，一个主节点对应一个或多个从节点。当其它主节点ping一个主节点A时，如果半数以上的主节点与A通信超时，那么认为主节点A宕机了，如果主节点宕机时，就会启用从节点。

6. 每个节点有：插槽（slot），取值范围是0~16384；另外一个是cluster，可以理解为一个集群管理的插件

7. 对于客户端来说，整个集群cluster，被看作一个整体，连接任何一个node，看起来和操作单实例redis一样。

8. 集群内部节点出现故障时，通过故障转移，以保证集群正常对外提供服务。redis集群通过ping/pong消息，实现故障发现，这个环节包括主观下线和客观下线。
  - 主观下线：某个节点认为另外一个节点不可用，即下线状态。
  - 客观下线：指标记一个节点真正下线，集群内多个节点都认为该节点不可用，从而达成共识的结果。
  - 如果是持有槽的主节点故障，需要为该节点进行故障转移。
  - 假如节点A标记节点B为主观下线，一段时间后，节点A通过消息把节点B的状态发到其它节点，当其它节点接到消息，如果发现节点B的fail状态时，会触发客观下线流程。
  - 当下线节点为主节点时，此时集群为统计持有槽的主节点投票，票数达标时，标记为客观下线状态。
  - 故障恢复：故障发现后，如果下线节点是主节点，则需要在它的从节点中选一个替代它，以保证集群的高可用。
