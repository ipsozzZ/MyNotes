### # 软件生命周期
需求分析、软件设计、软件开发、运行维护、淘汰

为了使软件生命周期中的各项任务能够有序地按照规程进行，需要一定的工作模型对各项任务给予规程约束，这样的工作模型被称为软件过程模型，有时也称为软件生命周期模型。


### # 软件过程模型

 #### 瀑布模型（适合需求明确的中小型项目）
 是最早使用的软件过程模型之一，包含一系列活动。这些活动从一个阶段到另一个阶段逐次下降，它的工作流程在形式上很像瀑布，因此被称为瀑布模型。

 需求分析 -> 系统设计 -> 程序设计 -> 编码实现 -> 单元测试 -> 集成测试 -> 系统测试 -> 运行维护

 1. 特点：因果紧密相连，前一个阶段工作的输出结果，是后一个阶段工作的输入，每一个阶段都是建筑在前一个阶段正确实施的结果之上，每一个阶段工作完成后都伴随着一个里程碑（一组检查条件），对该阶段的工作进行审查和确认。

 2. 主要缺点：(返工代价很大)
 - 软件需求的完整性、正确性等很难确定，甚至是不可能和不现实的。
 - 是一个严重串行化的过程模型，使得用户和软件项目负责人要相当长的时间才能得到一个可以看得见的软件系统。
 - 基本原则是在每一个阶段一次性地完全解决该阶段的工作，不会出现遗漏、错误等情况，而实际上这是不现实或不可能的。
 
 #### 原型模型（Prototype Model），又称快速模型

(需求分析 -> 系统设计 -> 程序设计) -> 编码实现 -> 单元测试 -> 集成测试 -> 系统测试 -> 运行维护
  |
原型开发

原型模型主要有以下两个阶段：
1. 原型开发阶段。即快速开发一个原型；开发原型可以考虑3种途径：利用模拟软件系统的人机界面和人机交互方式；真正开发一个原型；找来一个或几个正在运行的类似软件进行比较。
2. 目标软件开发阶段。在征求用户对原型的意见后对原型进行修改完善，确认软件系统的需求并达到一致的理解，进一步开发实际系统。

3. 原型模型的使用应该注意以下内容。
- 用户对系统的认识模糊不清，无法准确回答目标系统的需求。
- 要有一定的开发环境和工具支持。
- 经过对原型的若干次修改，应收敛到目标范围内，否则可能会失败。
- 对大型软件来说，原型可能非常复杂而难以快速形成，如果没有现成的原型模型，就不应该考虑用原型法。

4. 演化：按作用不同出现了抛弃型原型和演化性模型。

#### 螺旋模型（Spiral Model）（大型、高风险项目）
是在快速原型的基础上扩展而成，实际上它是生命周期模型与原型模型的结合。自己搜图

1. 这种模型把整个软件开发流程分成多个阶段，每一个阶段都由4个部分组成，它们是：目标设定、风险分析、开发和有效性验证、评审。

2. 螺旋模型的软件开发过程实际是上述4个部分的迭代过程，每迭代一次，螺旋线就增加一圈，软件系统就生产一个新版本，这个新版本实际上是对目标系统的一个逼近。

3. 该模型支持大型软件开发，适用于面向规格说明、面向过程和面向对象的软件开发方法，也适用于几种开发方法的组合。

#### V模型
从整体上看起来，就是一个V字型的结构，由左右两边组成。左边的下画线分别代表了：需求分析、概要设计、详细设计、编码。右边的上画线代表了：单元测试、集成测试、系统测试、验收测试。特点如下：
1. 单元测试的主要目的是针对编码过程中可能存在的各种错误；（单元测试 测 编码）
2. 集成测试的主要目的是针对详细设计中可能存在的问题；（集成测试 测 详细设计）
3. 系统测试主要针对概要设计，检查系统作为一个整体是否有效地得到运行；（系统测试 测 概要设计）
4. 验收测试通常由业务专家或者用户进行，以确认产品能真正符合用户业务上的需要。（验收测试 测 需求）
5. V模型用于需求明确和需求变更不频繁的情形。
6. 特别注意：测试阶段依赖的文件（注意和上面测的内容不一样，而是要依赖于上一个阶段）是对应开发阶段的上一个阶段，如单元测试的依据是详细设计文件。（对应开发阶段的同一行的下一行的文件）


#### 增量模型
首先开发核心模块功能，而后与用户确认，之后再开发次核心模块的功能，即每次开发一部分功能，并与用户需求确认，最终完成项目开发，优先级最高的服务最先交付。由于并不是从系统整体角度规划各个模块，因此不利于模块划分。难点在于如何将客户需求划分为多个增量。与原型不同的是增量模型的每一次增量版本都可作为独立可操作的作品，而原型的构造一般是为了演示。

#### 喷泉模型
是一种以用户需求为动力，以对象作为驱动的模型，适合于面向对象的开发方法。使开发过程具有迭代性和无间隙性。

#### 基于构件的开发模型CBSD
利用预先包装的构件来构造应用系统。构件可以是组织内部开发的构件，也可以是商品化成品软件构件。

特点：增强了复用性，在系统开发过程中，会构件一个构件库，供其它系统复用，因此可以提高可靠性，节省时间和成本。

#### 形式化方法模型
建立在严格数学基础上的一种软件开发方法，主要活动是生成计算机软件形式化的数学规格说明。

### # 敏捷模型
开发宣言：个性和交互胜过过程和工具、可以工作的软件胜过面面俱到的文档、客户端合作胜过合同谈判、响应变化胜过遵循计划。

1. 敏捷方法区别于其他方法的两个特点
- 是“适应性”而非“预设性”。
- 是“面向人的”而非“面向过程的”。

2. 敏捷方法的核心思想：
- 敏捷方法是适应型，而非可预测型。拥抱变化，适应变化。
- 敏捷方法是以人为本，而非以过程为本。发挥人的特性。
- 迭代增量式的开发过程。以原型开发思想为基础，采用法代增量式开发，发行版本小型化。

#### 主要敏捷方法
1. 极限编程（XP）。基础和价值观是交流、朴素、反馈和勇气，即任何一个软件项目都可以从4个方面入手进行改善：加强交流；从简单做起；寻求反馈；勇于实事求是。
- XP是一种近螺旋式的开发方法，它将复杂的开发过程分解为一个个相对比较简单的小周期：通过积极的交流、反馈以及其他一系列的方法，开发人员和客户可以非常清楚开发进度、变化、待解决的问题和潜在的困难等，并根据实际情况及时地调整开发过程。

### # 需求工程
软件需求包括3个不同的层次：
1. 业务需求：反映了组织机构或者客户对系统、产品高层次的目标要求。
2. 用户需求：描述了用户使用产品必须要完成的任务，是用户对该软件产品的期望。这两种构成了用户原始需求文档的内容。
3. 功能需求（也包含非功能需求），给开发人员使用的需求文档：规定了开发人员必须在系统中实现的软件功能，用户利用这些功能来完成任务，满足业务需求。所谓特性是指逻辑上相关的功能需求的集合，给用户提供处理能力并满足业务需求。
4.作为补充，软件需求规格说明还应包含非功能需求，它描述了系统展现给用户的行为和执行的操作等。它包含产品必须遵从的标准、规范和合约；外部界面的具体细节；性能要求；设计或实现的约束条件及质量属性。所谓约束是指对开发人员在软件产品设计和构造上的限制，常见的有设计约束和过程约束。质量属性是通过多种角度对产品的特点进行描述，从而反映产品功能。多角度描述产品对用户和开发人员都极为重要。

### # 结构化分析
前面说结构化开发方法（**自顶向下**，逐层分解，逐步求精，**面向数据流**，上层是下层的输入）

#### 系统分析阶段概念（建立系统的逻辑模型，形成系统需求规格说明书）
1. 主要手段：数据流图、数据字典、结构化语言、判定表以及判定树等

2. 结构化分析的步骤：
  - 1.分析业务情况，做出反映当前物理模型的数据流图（DFD）
  - 2.推导出等价的逻辑模型DFD
  - 3.设计新的逻辑系统，生成数据字典和基元描述
  - 4.建立人机接口，提出可供选择的目标系统物理模型的DFD
  - 5.确定各种方案的成本和风险等级，据此对各种方案进行分析
  - 6.选择一种方案
  - 7.建立完整的需求规约

前四个步骤是：有一个已经存在的物理模型数据流图（DFD）->推导出逻辑数据流图（DFD）->推导出新的逻辑数据流图（DFD）->推导出新系统的物理模型数据流图（DFD）
第五步确定设计方案，第六步选择方案，第七步建立完整的需求规约。

3. 结构化分析的三大模型：
  - 功能模型（数据流图，反映系统有哪些功能，功能走向，业务流程等）（结构化分析的主要考点）
  - 行为模型（状态转换图（STD），系统有哪些状态、状态之间是怎么切换的）
  - 数据模型（实体联系图（E-R图）在数据库中已经讲过）
  - 数据字典




#### 结构化分析-数据流图DFD（用来对功能建模）
数据流图（DFD）：DFD需求建模方法，也称过程建模和功能建模方法。DFD建模方法的核心是数据流，从应用系统的数据流着手以图形方式刻画和表示一个具体业务系统中的数据处理过程和数据流。
  - 组成：DFD方法由4种基本元素（模型对象）组成：数据流、处理/加工、数据存储和外部项。
  - 图形符号：a.外部实体（是一个实体，图形是长方形）。b.处理/加工（是一个功能，图形：圆形或者四个角上是弧形的长方形）。c.数据存储（图形是：两条平行的线或者右边少一竖的且每条边粗细不一样的长方形）。d.数据流，使用箭头表示。

1. 数据流：使用箭头表示，箭头上标注的内容可以是信息说明或数据项。（注意其中一端必须要经过加工/处理）
2. 加工/处理：是一个功能，图形：圆形或者四个角上是弧形的长方形，表示对数据进行的加工和转换。指向处理的数据流为该处理的输入数据流，离开处理的数据流为该处理的输出数据。
3. 数据存储：两条平行的线或者右边少一竖的且每条边粗细不一样的长方形表示，表示用数据库形式或者文件形式存储的数据。
4. 外部项：图形是长方形，也称数据源或者数据终点，描述系统数据的提供者或者数据的使用者，如教师、学生、采购员、某个组织或部门或者其他系统等。

5. 数据流图中常见的三种错误：
  - 加工/处理有输入但是没有输出，称之为“黑洞”。
  - 加工/处理有输出但是没有输入，称之为“奇迹”。
  - 加工/处理输入不足以产生输出，称之为“灰洞”。
  - 错误可能的原因：一个错误的命名过程；错误命名的输入或输出；不完全的事实，灰洞是最常见的错误，也是最使人为难的错误。一旦数据流图交给了程序员，到了一个加工的输入数据必须足以产生输出数据流。
  - 一个存储必定有流入数据流和流出数据流。
  - 一个数据流至少有一端是加工/处理端


6. 分层数据流图
  - 顶层图：在顶层图中数据没有“加工/处理”，没有数据存储的，只展示外部实体之间的数据流。
  - 0层图：完整的数据流图。
  - 1层图，加工/处理图：对0层图里的“加工处理”进行分解而来，因为加工是一个功能（分解出一个子数据流图）。
  - n层图，加工/处理图：对n-1层图里的“ 加工处理”进行分解而来，因为加工是一个功能（分解出一个子数据流图）。
  - 注意父图描述过的数据流（输入、输出数据流）必须要在相应的


#### 结构化分析-数据字典（在建模过程中，数据流图中有不知道的符合或者内容，可以通过数据字典查询）
是一种用户可以访问的记录数据库和应用程序元数据的目录。目的是对数据流程的所有数据元素做出详细的说明，简而言之，数据字典是描述数据的信息集合，是对系统中使用的所有数据元素定义的集合。

1. 数据项：数据流图中数据块的数据结构中的数据项说明。数据项是不可再分的数据单位。
2. 数据结构：数据流图中数据块的数据结构说明，数据结构反映了数据之间的组合关系。
3. 数据流：数据流图中流线的说明。
4. 数据存储：数据流图中数据块的存储特征说明。
5. 处理过程：数据流图中功能块的说明。


#### 系统设计阶段（建立系统的物理模型主要内容包括：概要设计和详细设计）
架构化设计是一种面向数据流的设计方法，它以SRS和SA阶段所产生的数据流图和数据字典 等文档为基础，是一个自顶向下、逐步求精和模块化的过程。SD方法的基本思想是将软件设计成由相对独立且具有单一功能的模块组成的结构，分为概要设计和详细设计两个阶段。

1. 概要设计的主要任务是确定软件系统的结构，对系统进行模块划分，确定每一个模块的功能、接口和模块之间的调用关系（输出模块结构图）；

2. 详细设计阶段的主要任务是为每个模块设计实现的细节（包括模块内详细算法设计、模块内数据结构设计、数据库的物理设计、代码、输入/输出格式、用户界面等设计）。

3. 设计原则：
  - 信息隐藏与抽象：采用封装技术，将程序模块的实现细节隐藏起来，对外仅暴露接口。
  - 模块化：模块是实现功能的基本单位，它一般具有功能、逻辑和状态3个基本属性。模块的外部特征是指模块的模块名、参数表和给程序乃至整个系统造成的影响，而模块的内部特征性则是指完成其功能的程序代码和仅供该模块内部使用的数据。
  - 高内聚、低耦合

#### 系统设计阶段-内聚分类（从低到高）
1. 偶然内聚：一个模块内的个处理元素之间没有任何联系（无直接关联）
2. 逻辑内聚：模块内执行若干个逻辑上相似的功能，通过参数确定该模块完成哪一个功能（逻辑相似、参数决定）
3. 时间内聚：把需要同时执行的动作组合在一起形成的模块（为了同时执行组合在一起）
4. 过程内聚：一个模块完成多个任务，这些任务必须按指定的过程执行（指定的过程顺序）
5. 通信内聚：模块内所有处理元素都在同一个数据结构上操作，或者个处理使用相同的输入数据或者产生相同的输出数据（相同数据结构、相同输入或输出）
6. 顺序内聚：一个模块中的各个处理元素都密切相关于同一功能且必须顺序执行，前一个功能元素的输出就是下一个功能元素的输入（顺序执行、输入为上一个输出）
7. 功能内聚：最强的内聚，模块内的所有元素共同作用完成一个功能，缺一不可（共同作用，缺一不可）

#### 系统设计阶段-耦合分类（从低到高）
1. 非直接耦合： 两个模块之间没有直接的关系，它们分别从属于不同模块的控制与调用，不传递任何信息（无直接关系）
2. 数据耦合：两个模块之间有调用关系，传递的是简单的数据值，相当于高级语言中的值传递（传递数据值调用）
3. 标记耦合：两个模块之间传递的是数据结构（传递数据结构）
4. 控制耦合：一个模块调用另外一个模块时，传递的是控制变量，被调用模块通过该控制变量的值有选择的执行模块内的某个功能（控制变量、选择执行某一功能）
5. 通信耦合：一组模块共用了一组输入信息，或者它们的输出需要整合以形成完整数据，即共享了输入或输出（公用通信信息）
6. 公共耦合：通过一个公共数据环境相互作用的那些模块间的耦合（共享数据结构）
7. 内容耦合：当一个模块直接使用另一个模块的内部数据，或者通过非正常入口转入另一个模块内部（模块内部关联）

#### 系统设计阶段-图形工具
1. 系统结构图（又称模块结构图）
它是软件概要设计阶段的工具，反映系统的功能实现和模块之间的联系与通讯，包括模块之间的层次结构，即反映了系统的总体结构

2. 详细设计的基本步骤：
  - 分析并确定输入/输出数据的逻辑结构。
  - 找出输入数据结构和输出数据结构中有对应关系的数据单元。
  - 按一定的规则由输入、输出的数据结构导出程序结构。
  - 列出基本操作与条件，并把它们分配到程序结构图的适当位置。
  - 用伪代码写出程序。

3. 详细设计的表示工具有：图形工具、表格工具和语言工具。其中图形工具有业务流图、程序流程图、PAD图、NS流程图。

4. 程序流程图又称为程序框图。用方框表示一个处理步骤，菱形表示一个逻辑条件，箭头表示控制流向。其优点是结构清晰，易于理解，易于修改。缺点是：只能描述执行过程而不能描述有关的数据。

5. NS流程图，也称为盒图，是一种强制使用结构化构造的图示工具，也称为方框图。其具有以下特点：功能域明确、不可能任意转移控制、很容易确定局部和全局数据的作用域，很容易表示嵌套关系及模块的层次关系。

6. PAD图（专门适用于结构化设计阶段的）：是一种改进的图形描述方式，可以用来取代程序流程图，相比程序流程图更直观，结构更清晰。最大的优点是能够反映和描述自顶向下的历史和过程。PAD提供了5种基本控制结构的图示，并允许递归使用。


#### 系统设计阶段-人机界面设计（不重要，10年来考过3次）
1. 三大原则：
  - 置于用户控制之下
  - 减少用户的记忆负担
  - 保持界面的一致性


### 测试方法（论文种常考）

#### 测试分类
1. 根据程序是否运行分为：
  - 静态测试：只分析代码的语句、结构、过程等来检查程序是否有错误。方法包括：1.桌前检查（程序员自己检查，一般是代码完成后，单元测试前）；2.代码审查（若干程序员和测试组成的评审小组，通过召开程序评审会来进行审查）；3.代码走查（也是开会对代码审查，但是是有测试人员提供测试用例，让程序员扮演计算机的角色，手动运行测试用例来检查代码逻辑）
  - 动态测试：通过运行被测试程序，对得到的结果与预期的结果进行比较分析，同时分析运行效率和健壮性能等。分为三步：构造测试实例、执行程序以及分析结果。

2. 根据具体实现算法细节和系统内部结构的相关情况为根据可以分为：
  - 黑盒测试：根据需求规格说明书设计测试用例进行测试（功能测试）
  - 白盒测试：通过程序员了解代码结构，为代码每个语句、控制流程、函数模块等进行覆盖测试（如程序员的单元测试就是白盒测试法）
  - 灰盒测试：介于黑盒测试与白盒测试之间。灰盒测试除了重视输出相对于输入的正确性，也看重其内部的程序逻辑。但是不像白盒测试那样详细和完整。比白盒效率高，比黑盒实用性广。

3. 程序执行方式：
  - 人工测试
  - 自动化测试

4. 按测试阶段分类：
  - 单元测试（也称模块测试）测试依据是**软件详细设计说明书**。（用黑盒测试法验证功能，再用白盒测试验证覆盖率）
  - 集成测试：目的是检查模块之间，以及模块和已集成的软件之间的接口关系，验证已集成的软件是否符合设计要求。测试依据是**软件概要设计文档**。一般采用黑盒加白盒测试方法结合测试
  - 系统测试（测试依据是需求规格说明书）：一般情况下采用黑盒测试，本阶段主要包含功能测试、性能测试、健壮性测试、安装或反安装测试、用户界面测试、压力测试、可靠性及安全测试等。测试人员必须进行多轮回归测试。系统测试的结束标志是测试工作已满足测试目标所规定的需求覆盖率，并且测试所发现的缺陷已全部归零。
  - 性能测试（更详细）：负载测试和压力测试
  - 验收测试：最后一个测试阶段交付前测试，测外还应进行Alpha测试和Beta测试
  - 回归测试：测试的目的是测试软件变更之后，变更部分的正确性和对变更需求的符合性，以及软件原有的、正确的功能、性能和其它规定的要求的不损害性。

5. 其它测试
  - AB测试：A版本和B版本，同一时间维度，相识的目标人群去使用，收集结果（根据反馈选择一个版本）
  - web测试
  - 链接测试：是web应用系统的一个主要特征，它是页面之间切换和指导用户去一些未知地址页面的主要手段。
  - 表单测试：当用户通过表单提交信息的时候，检查表单提交的各个功能。


#### 测试用例设计
1. 黑盒测试用例设计
  - 等价类划分：有效等价类，把所有数据按照某种特性进行分类，在其中取一个测试即可。不符合特性的值称为无效等价类。（多个测试用例时不要同时测试超过1个无效等价类，因为超过了你不知道是哪个无效等价类引起的问题）
  - 边界值划分：将每类的边界值作为测试用例，边界值一般为范围的两端值以及在此范围之外的与此范围间隔最小的两个值。如：年龄范围为0-150，则边界值为0，150，-1，151四个。
  - 错误推测：没有方法，凭经验。
  - 因果图：由一个结果来反推原因的方法，具体结果具体分析，没有固定方法。

2. 白盒测试用例设计
  - 语句覆盖SC：代码中的语句都执行一遍，覆盖层级最低，因为执行了所有语句，不代表执行了所有的条件判断。
  - 判定覆盖DC：代码中所有的判断语句的条件的真假分支都要覆盖一次。（整体判定的真和假，不关注判定中的条件：如：`if a>1 || b>2 || c<0 {...}`，只关注整个if最后是真还是假，不关注a、b、c的条件的真假。）
  - 条件覆盖CC：针对每一个判定条件内的每一个独立条件都要执行一遍真和假。（只关注判定中的条件：如：`if a>1 || b>2 || c<0 {...}`，只关注a、b、c的条件的真假，不关注整体if的真假）
  - 条件判断组合覆盖CDC：同时满足判定覆盖和条件覆盖
  - 路径覆盖：逻辑代码中的所有可行路径都覆盖了，覆盖层级最高
  - 覆盖层级由低到高：语句覆盖->判定覆盖/条件覆盖->条件判断组合覆盖CDC->路径覆盖。（判定覆盖、条件覆盖的覆盖层级相同）

3. McCabe度量法：又称为环路复杂度，以程序流程图为基础，有三种计算公式：
  - 边数-顶点数+2
  - 封闭区域个数+1
  - 判定节点个数+1（推荐使用的方法）

### 系统运行与维护

#### 遗留系统
遗留系统是指任何基本上不能进行修改和演化以满足新的变化了的业务需求的信息系统，

1. 特点：
  - 不能完全满足要求
  - 技术已经过时
  - 维护工作十分困难
  - 基本没有文档，很难理解

2. 改造策略（考过论文）
  - 技术水平高，业务价值低：集成
  - 技术水平高，业务价值高：改造
  - 技术水平低，业务价值高：继承
  - 技术水平低，业务价值低：淘汰

#### 系统转换
指新系统开发完毕，投入运行，取代现有系统的过程

1. 三种转换计划：
  - 直接转换（风险高）
  - 并行转换：新系统和老系统并行运行一段时间（成本高）
  - 分段转换：分期分批逐步转换，是直接转换和并行转换的集合，将大型系统分多个子系统，将子系统逐步转换（耗时，需要处理好新老系统混合使用，协调接口等）

2. 数据转换与迁移
  - 系统切换前通过工具迁移
  - 系统切换前采用手工录入
  - 系统切换后通过新系统生成

#### 系统可维护性
系统的可维护性定义为维护人员理解、改正、改动和改进这个软件的难易程度

1. 评价指标：
  - 易分析性
  - 易改变性
  - 稳定性
  - 易测试性
  - 维护性的依从性（软件产品遵循与维护性相关标准或约定的能力）

2. 系统维护包括硬件维护、软件维护和数据维护，其中软件维护类型如下：
  - 正确性维护：发现bug并修复
  - 适应性维护：由于外部环境发生改变，被动进行软件修改和升级
  - 完善性维护：用户提出更多需求，增加功能
  - 预防性维护：对未来可能发生的bug进行预防性的修改

### 净室软件工程（事前数学论证）（过于理论，现实基本不会采用）
净室软件工程是一种应用数学与统计学理论以经济的方式生产高质量软件的工程技术，力图通过严格的工程化的软件过程达到开发中的零缺陷或者接近零缺陷。净室方法不是先制作一个产品，再去消除缺陷，而是要求在规约和设计中消除错误，然后以“净”的方式制作，可以降低软件开发中的风险，以合理的成本开发出高质量的软件。

在净室软件工程背后的哲学是：通过第一次正确地书写代码增量，并在测试前验证它们的正确性，来避免对成本很高的错误消除过程依赖。它的过程模型是在代码增量积聚到系统的过程的同时，进行代码增量的统计质量验证。它甚至提倡开发者不需要进行单元测试，而是进行正确性验证和统计质量控制。

净室软件工程（CSE）的理论基础主要是**函数理论和抽样理论**

1. 净室软件工程应用技术手段：
  - 统计过程控制下的增量式开发
  - 基于函数的规范与设计
  - 正确性验证（CSE的核心）
  - 统计测试和软件认证

2. 净室软件的缺点：
  - CSE太理论化，需要更多的数学知识，正确性验证的步骤比较困难且比较耗时。
  - CSE开发小组不进行传统的模块测试，这是不现实的。
  - CSE也会带有传统软件工程的一些弊端。

### 基于构件的软件工程（CBSE）
基于构件的软件工程是一种基于**分布对象技术**、强调通过**可复用性构件设计与构造软件系统**的软件复用途径。CBSE体现了“**购买而不是重新构造**”的哲学，将软件开发的重点从程序编写转移到了基于已有构件的组装。

1. 用于CBSE的构件应该具备以下特征：
  - 可组装性：对于可组装的构件，所有外部交互必须通过公开定义的接口进行，同时它还必须对自身信息的外部访问。
  - 可不部署性：构件必须自包含，可在特定环境下独立运行，构件总是二进制形式，无须在部署前编译。
  - 文档化
  - 独立性
  - 标准化：构件标准化意味着在CBSE过程中使用的构件必须符合某种标准化的构件模型。

2. 构件模型定义了构件的实现、文档化以及开发的标准，其包含的模型要素为：
  - 接口
  - 使用信息
  - 部署

3. 构件模型提供了一组被构建使用的通用服务，这种服务包括以下两种
  - 平台服务，允许构件在分布式环境下通信互操作。
  - 支持服务，这是很多构件需要的共性服务。例如，构件都需要的身份认证服务。

中间件（和构件是一样的概念）实现共性的构件服务，并提供这些服务的接口。

4. CBSE过程是支持基于构件组装的软件开发过程，这个过程的6个主要活动：
  - 系统需求概览
  - 识别候选构件
  - 根据发现的构件修改需求
  - 体系结构设计
  - 构件定制与适配
  - 组装构件创建系统

5. CBSE过程与传统软件开发过程不同点
  - CBSE 早期需要完整的需求，以便尽可能多地识别出可复用地构件
  - 在过程早期阶段根据可利用地构件来细化和修改需求。如果可利用地构件不能满足用户需求，就应该考虑由复用构件支持的相关需求。
  - 在系统体系结构设计完成后，会有一个进一步的对构件搜索及设计精化的活动。修改或者寻找备用构件等。
  - 开发就是将已经找到的构件集成在一起的组装过程。

### 基于构件的软件工程-构件组装
构件组装是指构件相互直接集成或者是专门编写的“**胶水代码**”将它们整合在一起来创建一个系统或者另一个构件的过程。

1. 常见的组装构件有以下3种方式
  - 顺序组装：按顺序组装。
  - 层次组装：一个构件直接调用自另一个构件所提供的服务时。
  - 叠加组装：两个或两个以上构件放在一起来创建一个新构件。

2. 构件组装的3种不兼容问题（通过编写适配器解决）（选择题经常出现）
  - 参数不兼容：接口每一侧的操作有相同的名字，但是参数类型或者参数个数不相同。
  - 操作不兼容：提供接口和请求接口的操作名不同。
  - 操作不兼容：一个构件的提供接口是另一个构件请求接口的一个子集，或者相反。
